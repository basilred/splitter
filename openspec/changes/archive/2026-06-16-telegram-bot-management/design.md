## Context

Текущий бот (Grammy) использует плоские команды и не имеет интерактивной навигации. Администратор-менеджер может только создать столик и получить QR-код. Для ежедневной работы нужно: открывать счёт, добавлять позиции (вручную и из шаблонов меню), отслеживать оплаты, закрывать счёт.

База данных — SQLite через Drizzle ORM. Сервисы (`TableService`, `BillService`, `PaymentService`, `VenueService`) уже поддерживают CRUD для счетов и платежей. Сессии бота хранятся в памяти процесса.

**Ограничения:**
- Бот работает в двух режимах: polling (dev) и webhook (prod через `/api/telegram-webhook`)
- Сессии в памяти — не подходят для горизонтального масштабирования, но для MVP приемлемо
- Все цены хранятся в копейках (целые числа)

## Goals / Non-Goals

**Goals:**
- Полноценная инлайн-навигация в боте с меню столиков, счетов и заведения
- Управление счетами: открыть, добавить/удалить/редактировать позиции, закрыть
- Шаблоны меню заведения: CRUD через бота
- Push-уведомления менеджеру о новых платежах
- Реальная статистика оплат
- Код, готовый к переезду на PostgreSQL (никакой SQLite-специфичной логики в боте)

**Non-Goals:**
- Веб-админка (остаётся как есть, с моковыми данными)
- Интеграция с POS-системами (только заглушки, как сейчас)
- Реальный платёжный шлюз
- E2E-тесты (unit-тесты сервисов и smoke-тест API достаточно)
- Сессии бота в Redis/БД (память процесса для MVP)

## Decisions

### 1. Инлайн-навигация: callback_data + стейт-машина в сессии

**Решение:** Использовать встроенные callback_query Grammy + сессию для хранения текущего контекста (что сейчас смотрит пользователь, ожидает ли бот ввода).

**Формат callback_data:** `namespace:action:payload`, где:
- `namespace` — раздел (table, bill, menu, venue, main)
- `action` — действие (show, add, delete, edit, back, close)
- `payload` — ID или доп. данные

Примеры: `table:show:uuid`, `bill:add_item:uuid`, `menu:add_category`

**Стейт-машина:**
```
IDLE ──→ просмотр столиков ──→ просмотр счёта
  │                              │
  └──→ ожидание ввода названия ←─┘
       (add_item_name, add_item_price, add_item_qty)
```

Сессия хранит:
```ts
interface SessionData {
  venueId?: string;
  context: 'idle' | 'viewing_tables' | 'viewing_bill' | 'viewing_menu' | 'awaiting_input';
  currentTableId?: string;
  currentBillId?: string;
  awaitingInput?: {
    type: 'add_item_name' | 'add_item_price' | 'add_item_qty' | 'add_item_from_menu';
    billId: string;
    data?: Record<string, unknown>;
  };
}
```

**Альтернатива:** Grammy Conversations plugin — даёт `enter()`/`exit()`, но требует дополнительной настройки и менее прозрачен. Для MVP проще самописная стейт-машина.

### 2. Модель данных для меню

**Решение:** Две таблицы — `menu_categories` (для группировки) и `menu_items` (шаблоны позиций). Категории необязательны (позиция может быть без категории).

```sql
menu_categories {
  id        text PK
  venue_id  text FK → venues.id
  name      text          -- "Кофе", "Чай", "Десерты"
  sort_order integer      -- порядок отображения
}

menu_items {
  id          text PK
  venue_id    text FK → venues.id
  category_id text FK → menu_categories.id (nullable)
  name        text          -- "Эспрессо"
  unit_price  integer       -- в копейках
  is_active   boolean       -- можно скрыть без удаления
  created_at  timestamp
  updated_at  timestamp
}
```

**Альтернатива:** Одна таблица `menu_items` с текстовым полем `category`. Проще, но без сортировки и переиспользования категорий. Две таблицы — правильнее с точки зрения нормализации и готовности к POS-интеграции.

### 3. Добавление позиции: из меню или произвольно

При добавлении позиции в счёт показываем выбор:
```
[📋 Из меню] [✏️ Своё название]
```

- **Из меню**: бот показывает категории → позиции → количество. Создаётся `bill_item` с name и unit_price из `menu_item`.
- **Своё название**: пошаговый диалог: название → цена (в рублях, бот конвертирует в копейки) → количество. Создаётся `bill_item` без привязки к `menu_item`.

### 4. Уведомления: вебхук от API-роута к боту

**Решение:** Когда гость оплачивает через `/payment/[id]`, API-роут вызывает `bot.api.sendMessage()` с уведомлением.

Так как бот и API живут в одном Next.js процессе, уведомления отправляются синхронно из того же приложения:

```
Guest pay → POST /api/payments → PaymentService.update()
                               → bot.api.sendMessage(chatId, "💵 ...")
```

**Альтернатива:** Polling от бота к БД — лишние запросы. Webhook от платёжного шлюза — избыточно для MVP.

### 5. Статистика: расширение PaymentService

**Решение:** Добавить метод `PaymentService.getStats(venueId, period)` с агрегацией по датам. Агрегацию делаем на уровне SQL (Drizzle raw query или sql-тег), так как данных может быть много.

### 6. Архитектура файлов бота

Рефакторинг: выносим обработчики из монолитного `index.ts` в модули:

```
src/lib/telegram-bot/
├── index.ts           # инициализация, middleware, старт
├── middleware.ts      # аутентификация (без изменений)
├── qr.ts             # генерация QR (без изменений)
├── router.ts         # диспетчеризация callback_data
├── handlers/
│   ├── main.ts       # главное меню, /start, /help
│   ├── tables.ts     # просмотр/управление столиками
│   ├── bills.ts      # управление счетами и позициями
│   ├── menu.ts       # управление меню заведения
│   └── stats.ts      # статистика
├── keyboards.ts      # фабрики inline-клавиатур
└── session.ts        # типы и утилиты сессии
```

## Risks / Trade-offs

| Риск | Последствие | Митигация |
|------|-------------|-----------|
| Сессии в памяти потеряются при рестарте | Пользователь сбросит контекст (вернётся в IDLE) | Для MVP ок. При переезде на PostgreSQL добавить `DrizzleAdapter` для сессий |
| Ошибки в callback_data (дубли, невалидные payloads) | Пользователь видит "неизвестная команда" | Валидация callback_data через Zod в router.ts |
| SQLite на проде (если не переедут на PostgreSQL) | Блокировки при конкурентных запросах | PostgreSQL в документации как requirement, SQLite только для dev |
| Токен бота скомпрометирован | Доступ к боту от имени заведения | Только через привязку telegramId в БД (уже есть) |
| Менеджер не видит уведомления | Пропустит оплату | Уведомления дублируются в список платежей в боте |
